# Plugin distribution (marketplace repo + release automation)

The Claude Code plugin is published to a **dedicated, private distribution repo**
that doubles as a Claude Code marketplace — separate from this source repo, so
"who can install the tool" is decoupled from "who can read the source."

- **Source repo:** `tidy-dev-team/tidy-ds-toolbox` (private — this repo)
- **Distribution repo:** `tidy-dev-team/tidy-ds-toolbox-plugin` (private)
- **Marketplace name:** `tidy-ds-marketplace` · **Plugin slug:** `tidy-ds`

The distribution repo's contents are generated — the assembled `dist-plugin/`
tree (marketplace manifest + `tidy-ds/` plugin tree + bundled `mcp/server.cjs`).
Never hand-edit it; every release overwrites it.

## One-time setup

1. **Create the empty private repo** (needs org permission this CI account lacks):

   ```bash
   gh repo create tidy-dev-team/tidy-ds-toolbox-plugin --private \
     --description "Claude Code plugin distribution (marketplace) for the Tidy DS Toolbox — generated by the release pipeline."
   ```

2. **Add a deploy token secret** to the *source* repo so the release workflow can
   push to the *distribution* repo (the built-in `GITHUB_TOKEN` is scoped to this
   repo only). Create a fine-grained PAT with **contents: read/write** on
   `tidy-ds-toolbox-plugin`, then:

   ```bash
   gh secret set PLUGIN_DEPLOY_TOKEN --repo tidy-dev-team/tidy-ds-toolbox --body "<token>"
   ```

   Until this secret exists, the release `publish-plugin` job still runs and
   assembles the plugin, but **skips the push with a warning** instead of failing
   the release.

3. **Seed the repo** once (so `/plugin marketplace add` works before the first
   tagged release):

   ```bash
   npm run build:plugin
   cd dist-plugin && git init -b main && git add -A \
     && git commit -m "release: tidy-ds plugin (seed)" \
     && git remote add origin git@github.com:tidy-dev-team/tidy-ds-toolbox-plugin.git \
     && git push -u origin main
   ```

## Release flow

`npm run release:patch|minor|major` bumps `package.json`, commits, and tags
`vX.Y.Z`. Pushing the tag triggers `.github/workflows/release.yml`, which already
builds the Figma bundle and GitHub release, and now also runs `publish-plugin`:

1. `npm run build:plugin` — bundles the MCP server, assembles the marketplace
   tree, and **syncs the plugin version from `package.json`** (so `/plugin update`
   detects the new release).
2. Pushes the assembled tree to `tidy-ds-toolbox-plugin` `main` and force-updates
   the matching `vX.Y.Z` tag.

Plugin, MCP server bundle, and Figma bundle therefore release together from one
tag, at one version.

## Consumer install

```
/plugin marketplace add tidy-dev-team/tidy-ds-toolbox-plugin
/plugin install tidy-ds@tidy-ds-marketplace
```

Update with `/plugin update`. See [`plugin-dev.md`](plugin-dev.md) for the
local-path dogfood loop and the manual Figma round-trip check.
